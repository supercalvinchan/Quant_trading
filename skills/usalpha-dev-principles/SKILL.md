# USalpha Development Principles

## 触发条件

- 需要在 `USalpha` 中继续开发，而不是只跑现有脚本。
- 需要新增或修改数据、因子、模型、回测、可视化、实验流程。
- 需要多人或多 agent 并行协作，并希望减少误改、误判和研究噪声。

## 配套清单

- 做代码评审、变更验收时，读取 `references/review-checklist.md`。
- 做新因子、新模型、新数据源、新回测口径实验前，读取 `references/experiment-checklist.md`。
- 这两份清单是本 skill 的执行面；本文件只定义原则，不替代逐项检查。

## 核心目标

把 `USalpha` 当作量化研究仓库，而不是只看回测曲线的脚本集合。所有开发默认服务于四件事：

1. 结果可复现
2. 结论可解释
3. 变更可归因
4. 失败可定位

## 原则

### 1. 先复现，再优化

- 在声称“改进”之前，先确认旧流程能稳定重跑。
- 任何性能对比都必须绑定明确的运行区间、股票池、参数和输出目录。
- 没有可重跑基线的优化，默认不成立。

### 2. 时间纪律高于一切

- 严禁未来函数、标签穿越、全样本归一化、按未来信息筛样本。
- 训练、验证、回测、展示日期必须分开思考，不能因为界面方便而混用。
- 任何新特征、新标签、新筛选逻辑，都先问一句：这个时点真的拿得到吗。

### 3. 一次变更只回答一个问题

- 每次提交尽量只改变一个研究假设或一个工程问题。
- 不把“数据清洗 + 模型替换 + 回测口径变化”混在一次改动里。
- 如果必须打包修改，要把每一层影响单独写清楚。

### 4. 基线必须活着

- 轻量、朴素、可解释的 baseline 不能被新实验悄悄替换掉。
- 新模型、新因子、新搜索流程必须能和当前 baseline 并排比较。
- 当复杂方案没有稳定优于 baseline 时，保留 baseline 作为默认路径。

### 5. 研究结论默认保守

- 不因为一次高 Sharpe、单窗口胜利、少量股票表现好，就声称发现 alpha。
- 先看样本数、覆盖率、IC 稳定性、回撤、是否依赖极少数日期或极少数股票。
- 任何“最好结果”都要怀疑是否来自参数搜索、样本偶然性或回退数据。

### 6. 回测是证据，不是装饰

- 不为好看的净值曲线修改评估口径。
- 回测改动必须明确说明：是研究逻辑变化，还是会计口径变化。
- 若暂时没有交易成本、滑点、换手约束，必须明确承认这是研究原型，不伪装成可交易结论。

### 7. 接口稳定优先于局部省事

- 优先保持现有数据契约稳定：行情面板、特征索引、预测输出、artifact 结构都不要随意漂移。
- 如果确实要改接口，先统一调用链，再一起改消费者，避免半旧半新状态。
- 不把临时分析字段偷偷塞进核心结构里长期存活。

### 8. 回退必须可见

- 本仓库允许 `synthetic_fallback`、`local_fallback` 一类降级路径，但降级不是成功。
- 任何 fallback 都必须显式记录在日志、summary 或产物里。
- 不允许把回退结果和真实市场结果混写成同一类结论。

### 9. 产物优先于口头描述

- 研究结果优先落到 `artifacts/`、结构化 json/csv、测试输出或可复跑命令。
- 口头说“效果更好”没有价值；必须能定位到具体产物和指标。
- 新增实验时，优先保留能支持复盘的最小元数据。

### 10. 偏向简单表达

- 因子、特征、标签、评分函数，优先使用本仓库已经存在的表达方式。
- 新抽象只有在减少重复、降低耦合、或澄清边界时才值得引入。
- 为了“看起来高级”而增加层级、配置或框架，通常是在制造维护成本。

### 11. 测试跟风险走

- 改纯展示层，做轻验证即可。
- 改数据契约、因子计算、时间切分、训练标签、回测统计，必须补针对性验证。
- 测试优先覆盖研究不变量：无未来泄漏、索引对齐、样本非空、指标口径稳定。

### 12. 协作时尊重边界

- 不重写别人已经在改的模块，除非任务明确要求。
- 多人并行时，按模块或文件边界划清责任。
- 如果发现别人的改动影响你的任务，优先兼容并说明，不先回退。

## USalpha 特定约束

- 这是研究原型仓库，默认优先级是正确性、可解释性、复现性，不是吞吐极限。
- 当前主干是“美股 OHLCV -> 526 因子 -> 截面预测 -> 多空回测”，新工作应尽量沿这条链路扩展，而不是绕开它另起一套体系。
- `artifacts/` 是结果边界；`config.py` 中的默认值是研究默认口径；这些都不应被随意破坏。
- 若引入更复杂模型、更多因子或新数据源，必须保留旧路径可运行，方便回退和比较。

## 开发时的默认动作

1. 先识别这次改动属于哪一层：数据、因子、模型、回测、可视化、实验编排。
2. 写清楚这次改动要验证的唯一主问题。
3. 检查是否引入时间泄漏、口径变化或隐式 fallback。
4. 只做与问题直接相关的最小修改。
5. 用可复跑命令和产物验证，不用主观描述替代。
6. 在总结里明确写出：改了什么、没改什么、怎么验证的、还有什么风险。

## 明确避免

- 用 in-sample 结果给策略背书。
- 为了让结果“跑通”而静默填补关键错误。
- 把 synthetic 数据结果和 live 数据结果混为一谈。
- 没有基线对照就宣称提升。
- 在一次提交里同时改变标签定义、特征定义和回测口径。
- 因为短期方便而破坏数据结构或 artifact 约定。

## 完成标准

当一次开发满足下面条件时，才算完成：

- 可以说明这次改动的单一目标。
- 可以用明确命令重跑。
- 可以指出结果产物位置。
- 可以解释结果是否来自真实数据还是 fallback。
- 可以说明是否触碰时间切分、标签或回测口径。
- 可以说清剩余风险，而不是只报告最好看的数字。
