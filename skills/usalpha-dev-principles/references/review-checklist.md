# USalpha Review Checklist

用于 code review、agent review、提交验收。目标不是评价代码风格，而是优先发现研究失真、接口漂移和不可复现风险。

## 1. 问题定义

- 这次改动是否只回答一个主问题？
- 说明里是否明确区分“修 bug / 重构 / 新实验 / 口径调整”？
- 是否能看出为什么要改，而不只是改了什么？

## 2. 变更边界

- 改动是否集中在相关模块，没有顺手改无关逻辑？
- 是否保留了现有主链路：数据 -> 因子 -> 模型 -> 回测 -> artifact？
- 是否避免把临时分析代码长期塞进核心模块？

## 3. 时间一致性

- 是否引入未来函数、标签穿越、未来日期筛选？
- 特征是否只使用当时可得数据？
- 训练、测试、预测、展示日期是否清楚分离？
- 是否存在全样本标准化、全样本排名、全样本阈值这类泄漏？

## 4. 数据契约

- 行情面板是否仍然满足 `index=datetime`、`columns=MultiIndex(instrument, field)`？
- 因子/预测索引是否仍为 `(datetime, instrument)`？
- 必要字段名是否稳定：`open/high/low/close/volume` 及派生字段？
- 新增字段是否有明确消费者，而不是悄悄污染公共结构？

## 5. 因子与标签

- 新因子表达是否可计算、可解释，而不是只追求复杂？
- 标签定义有没有变化？如果有，是否被明确标注为口径变化？
- 新特征是否与现有特征高度重复，只是换了名字或参数？
- 是否解释了覆盖率、缺失值和极端值处理方式？

## 6. 模型与评估

- 是否保留 baseline 作为对照？
- 新模型是否能和旧模型在同一数据窗口上比较？
- 指标是否同时报告样本数、IC、RMSE、回撤、胜率等必要上下文？
- 是否存在只挑最好结果展示，而隐藏失败结果的情况？

## 7. 回测口径

- 回测逻辑是否发生变化？如果变化，是否说明是研究逻辑还是统计口径变化？
- `top_quantile`、选股分桶、收益聚合方式是否被改动？
- 是否把没有交易成本/滑点的研究结果说成可直接交易？
- 是否混用了 synthetic 数据和 live 数据结果？

## 8. Fallback 与错误处理

- 是否新增了 fallback 路径？
- fallback 是否被显式记录，而不是静默吞掉异常？
- fallback 产物能否与真实数据产物区分？
- 错误处理是否保留了足够的定位信息？

## 9. 复现性

- 是否给出可重跑命令、配置或参数？
- 产物目录、命名和 summary 是否仍然稳定？
- 输出是否足够支持复盘：配置、指标、关键中间结果？
- 若结论依赖外部 API 或在线数据，是否说明了不稳定因素？

## 10. 测试与验证

- 是否根据风险补了相称的验证？
- 如果改的是数据/时间/回测逻辑，是否有针对性检查而不只是“脚本能跑完”？
- 是否验证了非空样本、索引对齐、数值范围、split 合法性？
- 是否明确哪些部分还没测到？

## 11. 协作质量

- 是否覆盖或冲掉了别人已有改动？
- 多人并行的文件边界是否清晰？
- 提交说明是否足够让下一位开发者接手？

## 12. 通过标准

一个改动至少满足以下条件，才适合通过：

- 能说清单一目标
- 没有明显时间泄漏
- 没有隐式口径漂移
- 输出可复跑、可定位
- fallback 透明
- 风险和未验证项被诚实写明
